The beginnings of canary.io

Canary is a distributed HTTP measurement network that is built upon open source components and produces open data. It is bare plumbing, nothing more, intended to be easy to reason about and to serve as a basic utility. It is a building block rather than a feature-filled product. It takes several measurements per second, and makes a five minute sliding window of those measurements available via a simple HTTP API.

Why?

The world has long lacked an independent point of reference for measuring infrastructure's top-level availability and performance. We wanted something that is not coupled to any specific company's infrastructure or workflow, being general enough that anyone else could build upon it.

The service

List all sites being monitored

We currently house our list of watched URLs in a simple JSON file stored on S3.

$ curl -L http://checks.canary.io
[
  {
    "id": "http-github.com",
    "url": "http://github.com"
  },
  {
    "id": "https-github.com",
    "url": "https://github.com"
  },
  {
    "id": "http-git-io",
    "url": "http://git.io/tVDpSw"
  },
  {
    "id": "api.canary.io-measurements",
    "url": "https://api.canary.io/checks/github.com-ssl/measurements"
  }

List measurements for a given site

Measurement details for a given site can be consumed by querying api.canary.io. Measurements are taken via the excellent and widely used libcurl library.

In this example, we'll grab the measurements that happened in the last second for https://github.com and pipe the results to the popular jq tool to pretty print the results:

$ curl -s https://api.canary.io/checks/https-github.com/measurements?range=1 | jq '.'
[
  {
    "total_time": 0.08905,
    "starttransfer_time": 0.082189,

You can tweak the rangeparameter as need be, the maximum being 300 seconds. If you leave off range, it will default to 10.

The software

Canary is currently composed of two custom software components, sensord and canaryd, both distributed under the MIT license. In a typical installation, multiple instances of canarydwill be scattered around the globe, all reporting into a load balanced set of canarydinstances.

While the ideas behind canary.io have been hashed out over a small number of years, the current implementation is very immature. There is a lot of room for improvement: pull requests are welcome.

Using the open data

A number of users have developed creative solutions for presenting canary data.

For inspiration check out Jeremy Green's dazzling ember dashboard and Rick Yoesting's spectacular hubot integration.
